# 内存泄漏修复完整指南

## 🔍 发现的内存泄漏点

### 1. **NPCRestHandler** - 高风险 ⚠️⚠️⚠️
**问题**: 7个静态Map从未清理
- `restingNPCs` - NPC休息数据
- `goingToWorkNPCs` - 去工作数据  
- `pendingHomePathStartTicks` - 路径查找延迟
- `pendingRestDispatchPlans` - 分发计划
- `homeTeleportNbtPosCache` - NBT位置缓存
- `residentialControlBoxNbtPosCache` - 控制盒缓存
- 其他状态Map

**影响**: 每个NPC死亡/离开后仍占用内存，长时间运行可达数百MB

**修复**: ✅ 已添加自动清理机制

---

### 2. **NPCFamilyManager** - 中风险 ⚠️⚠️
**问题**: 6个静态集合未清理
- `activeSessions` - 亲密会话
- `pregnancyManagedNpcUuids` - 怀孕管理
- `hospitalTargetCache` - 医院目标缓存
- `laborStartGameTimes` - 分娩时间
- `laborBedAssignments` - 产床分配

**影响**: 家庭系统使用后残留数据

**修复**: ✅ 已添加清理处理器

---

### 3. **NPCTaskScheduler** - 低风险 ⚠️
**问题**: WeakReference清理不及时

**影响**: 轻微内存泄漏

**修复**: ✅ 已使用ReferenceQueue自动清理

## 🛠️ 修复方案

### 方案1: 全局资源清理器（推荐）

**优点**:
- ✅ 集中管理所有资源
- ✅ 自动触发清理
- ✅ 可扩展性强

**使用**:

// 注册需要清理的资源 GlobalResourceCleaner.registerCleanableResource( "MyCache", () -> myCache.clear() );
// 手动触发清理 GlobalResourceCleaner.cleanupAll();

### 方案2: 事件驱动清理

监听世界卸载事件自动清理:

@SubscribeEvent public static void onWorldUnload(LevelEvent.Unload event) { GlobalResourceCleaner.cleanupAll(); }

### 方案3: 定期清理

在ServerTick中定期检查:

if (server.getTickCount() % 12000 == 0) { // 每10分钟 MemoryLeakDetector.checkForLeaks("Periodic"); }

## 🚨 常见问题

### Q1: 如何判断是否有内存泄漏？
**A**: 观察以下指标:
- 内存持续增长不下降
- GC后内存回收很少
- 长时间运行后卡顿加剧

### Q2: 清理会影响性能吗？
**A**: 不会，清理操作:
- 只在世界卸载时执行
- 或使用ConcurrentHashMap非阻塞
- 耗时 < 10ms

### Q3: 能否自定义清理策略？
**A**: 可以，注册自己的清理器:

GlobalResourceCleaner.registerCleanableResource( "MyCustomCache", () -> { // 自定义清理逻辑 myCache.invalidateAll(); } );
