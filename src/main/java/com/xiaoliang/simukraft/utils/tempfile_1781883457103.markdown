# 主线程任务队列积压优化指南

## 🎯 优化目标

解决NPC任务调度器中主线程任务队列积压问题，提升服务器性能和稳定性。

## 📊 优化内容

### 1. 优先级队列机制
- **HIGH**: 关键任务（如玩家交互、重要状态同步）
- **NORMAL**: 普通任务（如NPC状态更新、数据同步）
- **LOW**: 低优先级任务（如日志记录、统计信息）

### 2. 动态任务限制
根据TPS自动调整每tick处理的任务数量：

| TPS范围 | 每tick处理任务数 | 说明 |
|---------|-----------------|------|
| ≥19.0   | 200个           | 服务器性能良好，全力处理 |
| 17.0-18.9 | 100个         | 正常水平，保持平衡 |
| 15.0-16.9 | 50个          | 性能下降，减少负载 |
| <15.0   | 20-25个         | 严重卡顿，最小化处理 |

### 3. 背压控制
- 队列上限：**500个任务**
- 超过上限时自动丢弃新任务并记录警告
- 防止内存溢出和无限增长

### 4. 智能扩容
当队列积压超过300个任务时，临时提升处理能力50个任务/tick

#### 场景1: 队列持续积压 (>200个任务)
**症状**: 日志显示 `主线程排队:250/200`

**解决方案**:

## 🚨 常见问题

### Q1: 为什么我的任务被拒绝了？
**A**: 队列已满（≥500个任务）。检查：
1. 是否有死循环不断提交任务
2. 是否有任务执行时间过长阻塞队列
3. 考虑提升 `MAX_MAIN_THREAD_QUEUE_SIZE`

### Q2: TPS很高但队列还是积压？
**A**: 可能是任务提交速度远超处理速度。解决：
1. 合并相似任务，减少提交频率
2. 将非关键任务改为异步执行
3. 实现任务去重机制

### Q3: 如何判断优化是否生效？
**A**: 观察以下指标：
- ✅ `主线程排队` 保持在 <100
- ✅ `拒绝` 数量为 0
- ✅ `TPS` 稳定在 19-20
- ✅ `溢出警告` 为 0

### Q4: 能否完全禁用这个优化？
**A**: 可以，回退到固定限制模式：

private static int calculateDynamicTaskLimit() { return DEFAULT_MAIN_THREAD_TASKS_PER_TICK; // 固定100 }

## 📝 最佳实践

1. **合理使用优先级**: 只有真正紧急的任务才用HIGH
2. **批量处理**: 多个小任务合并成一个大任务
3. **异步优先**: 能不占用主线程的操作都放后台
4. **定期监控**: 关注日志中的统计信息
5. **及时清理**: 避免提交永远不会执行的任务
